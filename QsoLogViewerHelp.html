Introduction

Ham radio operators use a number of different databases to manage their contacts, or QSOs. Many are web-hosted sites that are viewed by other hams and grant awards for achieving milestones such as all states contacted. Among these web-hosted sites three are quite popular, eqsl.cc, qrz.com and lotw.arrl.org . Operators generally use a PC based logging program, such as Ham Radio Deluxe (hrd) Logbook, N1MM Logger, Winlog32 or other programs to record their QSOs as they complete their contacts. Periodically hams export their recent QSOs to a standard file and upload it to one or more web-hosted sites.

Over time the web-hosted sites and the local logger(s) become inconsistent. Some of the reasons inconsistencies occur are:

  • hams forget to upload all of their QSOs

  • QSOs are uploaded twice. These duplicates should be caught by the logging site but they are not always discovered.

  • an operator has their clock improperly set to UTC. When the contacted operator upload their QSO log entry the two will not match, and a duplicate may be created.

QsoLogViewer is an application for aiding operators in finding duplicate QSOs and missing QSOs in these different repositories. Once duplicates are identified operator can use the edit features of the logbook to delete or correct those records. QsoLogViewer can also prepare an update file for missing QSOs so the operator can upload them into the correct logbook.

Important Note

QsoLogViewer is not an editor. You cannot edit your logbook database directly from the tables created by QsoLogViewer. Every logbook has different fields and conventions for assigning values to fields and storing them. Use QsoLogViewer to find duplicates and then correct them with the logbook tools.

QsoLogViewer can create difference files, that is a file that contains QSOs that exist in one logbook but are not found in another. That file can then be uploaded into the incomplete logbook and thereby synchronize the two logbooks. Use this feature with care. If you upload an incorrect difference file or load it into the wrong logbook you run the risk of creating a large number of duplicates.

QsoLogViewer does not operate directly on logbook databases, only on analyzes files downloaded from those logbooks. So QsoLogViewe can't corrupt the logbook database or induce errors in the logs unless you upload an invalid file.

Using QsoLogViewer

The following steps enable you to eliminate duplicate records and enter missing records into you logbooks.

Step 1 - Create a directory with downloaded AID files

You probably have multiple logbook databases, so create a new directory (folder) that will contain files from all logbooks you want to synchromize. I'll call that directory myLogs. Visit each web-hosted logbook and export (download) a complete file of your QSOs in ADI format. Make sure you download QSOs and not QSLs or you'll be missing a lot of records. ADIF is a standard for Amateur Digital Information Format. The other popular format, Cabrillo, is not supported by QsoLogViewer at this time. Use the export feature of your PC based logbook program to export a complete record of those databases to the same directory. Name the files to indicate where they came from. After downloading my directory had the following files:

qrz.adi eqsl.adi lotw.adi hrd.adi


Step 2 - Load and View an AID file

Select the first file you want to analyze. We'll use eqsl.adi in this example. Load the file using the menu items: File->Open. A file chooser dialog will let you select the file, eqsl.adi in our example. Once loaded you should see a table similar to the one below:


The table lists QSO records in the order received in the downloaded file. You can sort the records by clicking on the column headings. So to reorder the records by call sign click on the 'CALL' heading.

The columns shown when first opening the file are set by default to the ones shown above. You can modify the columns you'd like to see by selecting the Action->Edit Columns men item. You'll see a dialog similar to the one below.


You can hide a column by unchecking it, or show a column by checking it. The underlying QSO records are unaffected. Only the table view is affected.

Step 3 - Find Duplicates

To find duplicates in the adif file select the Action->Find Duplicates menu item. A dialog similar to the one below will appear.


Click OK and you will be taken back to the table. However now duplicate records are selected (highlighted). You can view only the duplicates by selecting the menu item View->Show Selected Records. Then the table will only show an updated list of selected records similar to the following.


In this example I clicked on the CALL header in the table to sort them by call sign. At this point you should log into your web-hosted or local logbook and delete the duplicate records. You may want to delete those records not acknowledged by the other operator, when possible.

Once you've corrected the source logbook you can download a revised AIDF file if the changes are extensive. That will prepare you for the next step, identifying missing records. Note that when you are only viewing selected records the QsoLogViewer title bar will include the term '- Updated List -'. This tells you that you are looking at selected records. To return to a view of the entire file use the
View->Show All Rows menu item.

Step 4 - Find and Upload Missing Records

To find duplicates in an adif file begin by opening the file you consider most complete with the

File->Open menu item. Then select the Action->Compare Files menu item. The file chooser menu will appear asking which file you want to compare to the open ADIF file. You may get a warning about duplicate records, but you can hit OK to continue. You should then see a dialog similar to the one below.



In this example eight missing records were found. The eight records are in the open file, qrz.adi, but not in the compare file, eqsl.adi. After you hit the OK button you'll be returned to the main table, showing qrz.adi in this example. At that point you will probably want to see only selected (highlighted) records. To see them use the View->Show Selected Rows menu item. You should see something like the following.


Look these records over and identify any you consider invalid and edit those out by selecting them (Cntrl+left click) and then using the menu item Edit->Delete. This action only deletes them from the Update List table; it does not affect the AIDF file or any logbook. You can now create an update file by selecting the File->Save menu item. I use the naming convention eqsl_update.adi so I am careful to only upload updates, not entire logbook AIDF files.

Once you have eliminated duplicates and uploaded missing files your logbooks should be synchronized.

Additional Menu Items

In addition to the menu items already covered, there are two that may be helpful.

Header Menu Item

After opening a file use the View->File Header menu item to display the opened AIDF file's meta data. While this can vary, it generally displays which logbook created the file, when it was created and which version of the AIDF standard it complies with. It will look something like the following.



Comparison Rules Menu Item

Qso log records are compared when searching for duplicates and when two logbook AIDF files are compared. By default QsoLogViewer looks for matches using Call Sign, Band, Qso Date and Qso Time On (the UTC time the Qso started). In using Qso Time On two times are considered identical if they are within a default of 5 minutes of each other. Users can change these criteria using the Action->Show Rules menu item. When selecting this menu item you will see the dialog below.



When using these fields QsoLogViewer ignores case, so the call sign KC1ATT matches with the call sign kc1att.

Check All States Menu Item

Users can check their progress toward logging QSOs for all fifty US states by selecting this menu item, after loading an ADIF log file. Both QRZ.com and LoTW include the home state, QTH, of operators in the schema for their databases. eqsl.cc does not include this information, so this menu item will not work with log files from that site. When selecting this item a dialog box, similar to the one below, will display the results.



Update Grid Info

Log files often having missing or incomplete gridsquare information. Gridsquare information is used to calculate the location of the other operator. Some logging programs retrieve gridsquare information in real time, as the call sign of the other operator is logged, but to use this feature you need to have a source. Two web sites that offer this service are qrz.com and hamqth.com . To use all the features of qrz.com you must be registered as an XML subscriber; currently that costs about $30 per year. The other site, hamqth.com is a free site, but I recommend users make a donation to help cover the cost of maintaining the site. To open the dialog below select 'Upgrade Grid Info' from the Action menu.


The dialog displays:

- the file you have opened for upgrading,

- the number of records in the file with grid information

- the number of records without grid information

To use the feature you must be a registered member of either hamqth.com or qrz.com . As mentioned earlier, there is no cost to register with hamqth.com . Registered users can then enter their user ID, generally your call sign, and password to the site. Then select the 'Update records' button. You may need to wait a few seconds for the session to begin but then records should be upgraded at a rate of at least one per second. QsoLogViewer will go through every entry and query the grid update site for relevant grid information. As the session progresses the 'No. recs updated' will show how many queries resulted in valid gridsquare information.

A pop up dialog will notify you when the update is complete. You then need to close the Update Grid Information dialog and then use the 'File->Save' menu item to save the updated file under a new name.

If QsoLogViewer fails to establish a session with the update site, you will see a dialog pop up giving the reason.
